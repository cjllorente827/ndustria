Pipeline Keyword Arguments
==========================

ndustria **"Pipelines"** have a number of kwargs that can help you configure your run.
By default all of the booleans are set to :code:`False`, but we can experiment with setting them :code:`True` in :file:`pipeline_kwargs.py`. 

.. code-block:: bash

    python pipeline_kwargs.py

name
----

The name kwarg can be used to :code:`name` the Pipeline you create. 
If left blank, it will be derived from the filename of the Pipeline's script.  

.. note:: It is best practice to always name your Pipeline so that you can easily use ndustria's command line interface to access the cache and log files.

..  code-block:: python

    pipe = Pipeline(name = "test_kwargs")

dryrun 
------

When True, :code:`dryrun` skips running Tasks but runs through the ndustria workflow creating log and cache files. 
This is used to test complex pipelines.

.. code-block:: python

    pipe = Pipeline(name = "test_kwargs", dryrun = True)
    
:code:`dryrun` makes all of the other kwargs not meaningful so make sure to set :code:`dryrun=False` to test the other kwargs. 


timeit
------

The timeit kwarg, when set to True, keeps track of wallclock time of each Task. 
These data will be output to a csv file in the cache and a quick and easy graph can be generated by running :code:`ndustria -t <name of script>` in the terminal 

We can add this to the script:

.. code-block:: python

    pipe = Pipeline(name = "test_kwargs", parallel = True, timeit = True)

Run in the terminal: 

.. code-block:: bash

    ndustria -t test_kwargs

.. note:: 
    
    If have already run the script before with and set :code:`timeit = True` this *does not* automatically :code:`rerun` all of tasks. 
    So if a task has already been cached you are timing the time to read the results from disk which should be essentially zero. 
    If you want to time each function "un-cached" you should set :code:`rerun=True` for the functions that you want to time.   

profiling
---------

ndustria utilizes the package [Line Profiler](https://kernprof.readthedocs.io/en/latest/) which can look at the time utilization line by line for Tasks. 
This has a higher over-head than :code:`timeit` but gives a lot more infomation. 
Line Profiler allows us to look at the time it takes line by line for a function to execute. 
Line Profiling can be turned on with :code:`profiling=True` and this creates a text file in the cache directory which can be accessed with :code:`ndustria -p <name of script>`

We can add this to the script: 


.. code-block:: python

    pipe = Pipeline(name = "test_kwargs", parallel = True, timeit = True, profiling = True)


Run in the terminal: 


.. code-block:: bash

    ndustria -p test_kwargs

memcheck
-------- 

memcheck, when set to True, collects initial, peak, and final memory usage of each Task. 
These data will be output to a csv file in the cache. 
Can have high overhead if you allocate a lot of small objects

We can add this to the script: 

.. code-block:: python

    pipe = Pipeline(name = "test_kwargs", parallel = True, timeit = True, profiling = True, memcheck = True)

Run in the terminal: 

.. code-block:: bash

    ndustria -m test_kwargs


parallel
--------

If True, the parallel kwarg uses a round robin approach to assign Tasks to multiple processes and runs them in parallel. 
To fully utilize the parallel kwarg we must execute the script with :code:`mpirun` and assign a number of processors to the script. 

For example if we set:


.. code-block:: python

    pipe = Pipeline(name = "test_kwargs", parallel = True)


We must also run: 

.. code-block:: bash

    mpirun -n 4 python pipeline_kwargs.py

    Any outputs (writing to file, creating graphs, etc) will be in their proper order once completed.
